What is git?

  1. It is a system that record changes to files over time.
  2. You can recall any specific version at any point of time.
  3. Many people can easily collaborate on a project and can have their own version of project file on their computer.

There are some terms :-

1. Repository

A repo is a container for a project you want to track with git (e.g a website project) . Now repo can be stored locally on your computer or remote which is an online service like github.

So if you want to start with a git , you need to tell to the git that what is the name of that folder that you want to make it a git project or say git repository.

Now to create a repository , you need to navigate into that folder (you can navigate by command  cd “path of the folder”) and then run the mentioned command.

Command to make a git repository :-

---> git init

after running this command a “.git file” will appear in your repository ,this “.git file” tells this is the repository which is being tracked by git and it is going to track all kind of changes that we make inside this project.

2. Commit (safe points)

You can think of commit as safe points

Suppose that you are playing a video game and then you come to a safe point such that if you are being killed in the game, you can continue the game again with that safe point.

Similarly suppose that you are creating a website and after creating some specific part of your website (like creating navigation bar, creating some kind of feature ) then after you have finished with that particular part of your project, you can commit your project that is like creating it a safe point such that if your code is messed up you can rewind and go back to any commit in the commit history and can see what the code looks like.

We have three stages for any project

Git working

Staging files

When we commit something ,all it does is take everything from staging area and create a commit from that.

Now so before commiting anything we need to add the file into the staging area.

Here is the question arises that why don’t we just commit straight away without adding changes to staging area?

Well, answer for this is that by adding them to the staging area it allow us to review our changes before we make that commit. We should commit at logical point after developing certain features of our website or application.

command to add files to the staging area.

---> git add filename

If there are more than 1 changes

---> Git add .

Commiting files

We can commit the changes as

---> git commit -m “a brief message to describe the commit”

To see the commit history

---> git log

To see the commit history in one line

---> git log –oneline

So how can we rewind things if we make mistake or messed up with our code

There are three methods:-

  • checkout - this is for read only

    you can see state of code at particular commit but can not make any changes there,you can checkout the particular commit as

    ---> git checkout commitUniqueId

    commitUniqueId is the five digit expression we see when we run the command “git log --oneline”.

    To go back , you need to run

    ---> git checkout master (see master is the branch , you don’t need to worry about this we will learn about branch after 1 minute.

  • revert - this makes another commit for undoing the specific commit.

    This is actually making a commit for deleting the specified commit , yes you can still visit that commit because we are not actually deleting the commit but we are making another commit for deleting the changes that we make in that specified commit.

    ---> git revert commitUniqueId

    this command will make another commit for deleting the changes that we make in the specified commit(commitUniqueId) at top of all the commits in the commit history.

  • reset - this reset or delete the commits after the specified commit.

    This commit is going to potentially destroy the contents of your repository so be really careful before doing this.

    Suppose that we have these commits in our commit history:

    F286814 – 4th commit
    B7b2d3d – 3rd commit
    2d63d76 – 2nd commit
    B946a4e – 1st commit

    See if we want to work from 2nd commit and we don’t want the commits after the 2nd commit, we can reset our commit history as:-

    ---> git reset commitUniqueId –hard

Branching

Master branch

When we create a new repository what we are doing is creating this master branch as well and when we have been making commits, we have been commiting to this master branch only.

This master branch is going to represent the stable version of your code and normally it is going to be the code which is released or published.

Now we don’t want to try out new feature or new code on this branch just in case it messes the code .So what we really want to do if we want to have new feature ,we create an isolated environment to try out new feature and merge them into master branch.

To create a branch :-

---> git branch branchName

Now if we want to see all the branches :-

---> git branch -a

To switch the branch :-

---> git checkout branchName

To delete a branch :-

git branch -d(or -D) branchName
-d -> if branch is merged.
-D -> if branch is not merged.

Merging branches

To merge any branch we have to be on the branch that we want to merge into (i.e the master branch)

And then just run the command :-

----> git merge branchName
